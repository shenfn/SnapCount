## TestFlight 节流协议

1. 普通 commit、修复 commit 和文档 commit 只触发 GitHub `iOS Build`，不得自动上传 TestFlight。
2. TestFlight 仅允许在以下情况手动触发：
   - 一个完整迁移阶段结束，需要统一真机验收；
   - 涉及 App Intent、快捷指令、Keychain、相机相册、通知、系统权限等无法通过 CI 验证的能力；
   - 用户明确要求生成真机版本。
3. 同一阶段的多个小 commit 必须合并为一次 TestFlight 验收，不得逐 commit 上传。
4. 上传前必须确认：远程最新提交 Build 成功、iOS 工作区无未提交目标改动、现有其他 Agent 改动不会进入构建。
5. Apple 返回 `90382 Upload limit reached` 后，不得连续重试；至少等待 Apple 提示的完整时间窗口后再手动尝试。
6. 2026-07-13 手动上传运行 `29254501519` 已成功，证明限额解除；后续仍按低频协议执行。

## 每批交付规则

每个任务只有同时满足以下条件才可改为 ✅：

1. 与 PWA 当前行为一致，或差异已明确确认。
2. 使用真实 Supabase 数据，不新增固定业务数据。
3. 加载、空、失败、重试和权限状态可解释。
4. GitHub iOS Build 成功。
5. 日常 commit 只要求 GitHub iOS Build 成功；TestFlight 仅在一个完整阶段结束、需要真机能力验证、或用户明确要求时手动上传。
6. 用户完成截图或操作复核。
