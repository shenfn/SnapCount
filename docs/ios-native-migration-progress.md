## P1 Dashboard 快速刷新链路

### 已确认根因

- PWA 的消费、收入、通用记录和中转站四类请求并行完成后，会先发布结构化数据；图片签名、账期和域定义属于补充加载。
- iOS 虽然已经并行查询四类数据，但原实现仍串行申请每张图片的签名 URL，并在图片和 `data_domains` 返回前阻塞整个 Dashboard 发布。
- 因此记录金额和标题已经从 Supabase 返回，用户仍会看到首页加载；图片越多，等待越明显。

### 本轮实现

### 真机验收

1. 冷启动存在本地快照时，先显示上次数据，再在四类核心请求返回后更新金额、数量和记录列表。
2. 下拉刷新不再等待图片签名和数据域定义才结束加载状态。
3. 同一会话重复刷新时，已经显示的图片不应先消失再出现。
4. 新增带图记录后，文字和金额应先出现，图片随后自动补上；补图失败不得清空核心数据。
5. 快速切换页面或连续触发刷新时，旧快照不得覆盖较新的记录。

### 后续边界

- 本轮是“远端结构化数据优先发布”，不是完整的本地优先数据库；四张业务表首次同步仍需要网络。
- 首次出现的图片仍需签名和下载，但不再阻塞首页，重复图片继续命中签名缓存与 `URLCache`/图片缓存。
- 若要达到成熟记账软件的近乎瞬时进入与离线可用，后续 B2/B4 仍需实现分域只读本地缓存、增量同步和本地原图。

## TestFlight 节流协议

1. 普通 commit、修复 commit 和文档 commit 只触发 GitHub `iOS Build`，不得自动上传 TestFlight。
2. TestFlight 仅允许在以下情况手动触发：
   - 一个完整迁移阶段结束，需要统一真机验收；
   - 涉及 App Intent、快捷指令、Keychain、相机相册、通知、系统权限等无法通过 CI 验证的能力；
   - 用户明确要求生成真机版本。
3. 同一阶段的多个小 commit 必须合并为一次 TestFlight 验收，不得逐 commit 上传。
4. 上传前必须确认：远程最新提交 Build 成功、iOS 工作区无未提交目标改动、现有其他 Agent 改动不会进入构建。
5. Apple 返回 `90382 Upload limit reached` 后，不得连续重试；至少等待 Apple 提示的完整时间窗口后再手动尝试。
6. 2026-07-13 手动上传运行 `29254501519` 已成功，证明限额解除；后续仍按低频协议执行。
