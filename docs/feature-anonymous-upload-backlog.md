# 待跟进需求：匿名上传功能

> 记录时间：2026-06-27
> 来源：域隔离 bug 修复讨论中引出
> 优先级：待评估
> 状态：需求记录，未开始设计

## 背景

当前用户使用 AI 截图识别功能前，需要手动将 upload_token 复制到 iOS 快捷指令中，配置成本高、体验差。由于 App 尚未上架 AppStore，这个前置配置环节对新用户体验影响较大。

匿名上传功能可以让新用户先不注册、不配置 token，直接体验截图识别功能，降低使用门槛。体验满意后再引导注册和配置。

## 需求方向

用户在未注册/未配置 token 的情况下，可以直接上传截图进行 AI 识别，体验核心功能。

## 已讨论的边界问题

### 1. 数据归属和可见性
- 匿名用户的数据 user_id 为 null，App 登录后因 RLS 策略看不到
- 需要决定：匿名数据是否入库？还是只返回识别结果给客户端？
- 如果入库，注册后如何关联到新账号？

### 2. 成本控制
- AI 识别每次调用有成本（通义千问 API）
- 当前没有配额检查逻辑（user_configs 有 monthly_quota/daily_quota 字段但代码未检查）
- 匿名用户需要 IP 限频？设备限频？还是给一个很小的试用额度？

### 3. 功能范围
- AI 识别：应该可以（核心体验）
- companion 文案：给默认的
- 数据归档到 data_records：匿名用户要不要入库？
- AI 反馈（状态签）：可以给
- 去重检查：匿名用户之间要不要做去重？

### 4. 入口设计
- 完全不传 user_id 和 upload_token 时走匿名模式
- 还是传一个明确的 `anonymous=true` 标记？
- 匿名模式的数据保留期？自动清理策略？

## 与当前 bug 修复的关系

当前域隔离 bug 修复（任务 4）已采用"报错拒绝"策略：传了无效身份信息返回 401，不做静默降级。匿名上传作为独立需求时，在 401 校验逻辑之前加入匿名模式的专门处理分支。

## 下一步

等当前 bug 修复完成后，作为独立需求出 PRD 和技术方案，讨论清楚上述边界问题后再实现。
