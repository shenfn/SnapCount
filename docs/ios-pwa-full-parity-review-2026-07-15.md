# 芥子 PWA -> iOS 原生功能完整性复审

> 日期：2026-07-15  
> 业务基线：`src/` 成熟 PWA、现有 Supabase 表/RPC/Edge Function  
> iOS 基线：`codex/ios-swiftui-native-app`  
> 目标：先迁移影响主流程的功能，视觉细节和真机个案进入后续修复批次。

## 1. 复审结论

当前 iOS 已具备可运行的主框架，图片上传识别、记录读取、账户财务、分析和快捷指令链路均已建立，但还不能定义为“PWA 主要功能已经全部迁移”。

静态代码核查共拆出 42 个功能点。完成 M1-M3 后：

- 已实现：31 项；
- 部分实现：9 项；
- 未实现：2 项。

当前最影响普通用户闭环的三个缺口：

1. 手动新增支出、收入和通用数据域已完成，通用记录编辑也已补齐；
2. 邮箱注册已完成，手机号/Apple 等登录不是当前 PWA 迁移基线；
3. 首页五组件和真实账户财务已完成，账期级最近待还仍需真机后继续细化。

这三项属于主功能，不应推迟到视觉打磨阶段。

## 3. 状态流与风险

### 3.1 手动记录

- 新建支出/收入必须继续调用 `save_transaction_with_account` / `save_income_with_account`，`p_id = null`，不能绕过账户流水 RPC；
- 新建通用记录必须先读取真实 `data_domains`，再写 `data_records`；域不存在时不得自动改写为其他 key；
- 保存成功后统一刷新 Dashboard、账户与详情缓存；保存失败不得先关闭表单；
- 金额、日期、必填维度和数值上限沿用 PWA 校验。

### 3.2 注册

- 注册成功且 Supabase 返回 session 时，继续自动同步 upload token；
- 邮箱确认开启且没有 session 时，保留在登录页并提示用户查收邮件；
- 注册失败不清空邮箱，密码不持久化。

### 3.3 首页组件

- 组件偏好只保存在当前设备，不写业务数据库；
- 固定五个 key：`finance`、`today`、`pending`、`domains`、`daily`；
- 旧版本没有偏好时使用 PWA 默认顺序；非法或缺失 key 自动归一化；
- 财务组件只消费账户、账期和 Dashboard 真实数据，不在客户端制造余额事实。

## 4. 实施顺序

### M1 主记录闭环

1. 增加手动记录数据契约和 Repository create 方法；
2. 增加支出、收入和通用域原生表单；
3. 从首页和记录页提供统一新增入口；
4. 保存后刷新相关数据并增加契约测试。

### M2 认证完整性

1. Supabase Auth 增加 signUp；
2. 登录页增加登录/注册分段模式；
3. 处理邮箱确认和 upload token 尚未生成的状态。

### M3 首页主要能力

1. 迁移五组件显示/隐藏/排序；
2. 财务状态接入真实账户、欠款和账期；
3. 增加待处理和数据域摘要组件；
4. 保留现有图片上传和日详情导航。

### 后续重要批次

- E3：设置数据导出、图片留存、协议和账号删除；
- C5：通用域编辑器、数据域趋势/分布；
- C6：中转站批量操作；
- B2/B4：分域只读缓存与本地原图。

## 5. 真机验收用例

### TC-001 手动新增支出
